# ツール設計書 — 動画簡易変換ツール

| 項目 | 内容 |
|------|------|
| ツール名 | 動画簡易変換ツール |
| バージョン | v0.3.1 |
| 作成日 | 2026-05-23 |
| 対象利用者 | 庁内一般職員 |
| 利用環境 | Windows 10/11、閉域または庁内ネットワーク |

---

## 1. 目的

一般職員が専門的な動画編集ソフトを使わずに、MP4動画の切り出し・サイズ縮小を行えるようにする。
庁内共有や資料添付に適した動画ファイルを、職員自身で簡単に作成できることを目指す。

---

## 2. 入力・処理・出力

| 区分 | 内容 |
|------|------|
| 入力 | MP4ファイル（ファイル選択またはドラッグ＆ドロップ） |
| 処理 | ffmpeg.exe による範囲切り出し・再エンコード |
| 出力 | MP4ファイル（入力ファイルと同じフォルダ、タイムスタンプ付きファイル名） |

---

## 3. 保存するデータ / 保存しないデータ

| 種別 | 内容 |
|------|------|
| 保存する | 変換後のMP4ファイル（出力先フォルダ） |
| 保存しない | 入力ファイルの内容、設定情報、ログ、個人情報 |
| 一時ファイル | WebView2のユーザーデータ（%TEMP%\MovieConverter_WebView2）※アプリ終了後も残る場合あり |

## 4. 技術選定と判断理由

### 4.1 フレームワーク: .NET 8 + WinForms

**選定理由:**

- WinForms は WPF と比較して実装容易性・保守性が高く、小規模庁内ツールに適している
- .NET 8 は LTS（長期サポート版）であり、モダンな C# の恩恵を受けられる
- WPF は XAML の習熟が必要で、保守担当者が変わった場合のリスクが高い

**トレードオフ:**

- .NET 8 ランタイムのインストールが必要（またはセルフコンテインド発行）
- .NET Framework 4.8 は既定でインストール済みだが、WebView2の統合が煩雑になる

**対応方針:**

配布時は `dotnet publish -r win-x64 --self-contained true -p:PublishSingleFile=true` でセルフコンテインドの単一実行ファイルとして配布することを推奨する。

---

### 4.2 動画プレビュー: WebView2 + HTML5 video

**選定理由:**

- HTML5 の `<video>` タグは MP4/H.264 を標準でサポートし、シーク・再生・一時停止を安定して提供する
- WebView2（Microsoft Edge WebView2 Runtime）は Windows 10 20H2 以降・Windows 11 に標準インストール済み
- C# ↔ JavaScript の双方向通信（PostWebMessage）により、再生位置・継続時間を正確に取得できる
- ローカルHTMLのみを使用し、外部CDNやネットワーク接続に依存しない
- WinForms に WebView2 コントロールとして埋め込めるため、1画面完結の構成を維持できる

**代替方式（WPF MediaElement）との比較（v0.3.1 で代替プレビューとして実装）:**

| 比較項目 | WebView2 + HTML video | WPF MediaElement |
|----------|----------------------|-----------------|
| プロジェクト方式 | WinForms | WPF + ElementHost（UseWpf=true） |
| MP4対応 | ○ | ○（コーデック依存） |
| シーク精度 | ○ | △（一部形式で不安定） |
| 保守性 | HTML/JS + C# | C#（XAMLなし） |
| 配布容易性 | WebView2 Runtime 必要 | .NET WPF ランタイムのみ |
| v0.3.1 での位置付け | 標準プレビュー | 代替プレビュー（フォールバック） |

**WMP COM（Windows Media Player ActiveX）との比較:**

WinForms で AxWindowsMediaPlayer を使う方法もあるが、.NET 8 + SDK スタイルプロジェクトでの COM 相互運用設定が煩雑になるため、標準方式としては採用しなかった。v0.4.7 にて検証目的で `WmpVideoPlayer` として実装済み（後述 4.6 節）。

---

### 4.3 動画処理: ffmpeg.exe 外部プロセス呼び出し

**選定理由:**

- **ライセンスの明確性**: FFmpeg DLL を直接リンクする場合、GPL/LGPL の条件（ソースコード開示義務等）がより厳格に適用される可能性がある。外部プロセス呼び出し（プロセス境界での分離）は、ライセンスリスクを相対的に低減する
- **実装の安全性**: DLL を直接呼び出す場合、FFmpeg のネイティブ API に精通している必要があり、初期版の開発コストが高い
- **安定性**: 外部プロセスとして呼び出すことで、FFmpeg クラッシュがアプリ本体に影響しない
- **配布の容易性**: ffmpeg.exe を bin/ffmpeg/ フォルダに置くだけで動作し、管理者が配置作業を理解しやすい

---

### 4.4 ビューワはFFmpegではなくアプリ側で実装する理由

- FFmpeg はコマンドラインツールであり、リアルタイム動画プレビューには適していない
- WebView2 の HTML5 video タグは OS のメディア機能を活用し、再生・シーク・一時停止を軽量に実現する
- FFmpeg でプレビュー用のフレームを逐次生成する方式は実装が複雑で、レスポンスが悪い

---

### 4.5 WebView2 プレビュー読み込み方式（v0.1.2改訂）

**確認された事象:**

実機確認で、AVC/H.264・AAC・MP4コンテナの動画（isom/iso2/avc1/mp41、High@L3.1、1280×720、30fps、約3時間20分、約1.55GiB）を読み込んだところ、WebView2のプレビューで再生できないケースが発生した。同じ動画をMicrosoft Edgeで直接開くと再生できた。

**原因分析:**

AVC非対応ではなく、v0.1.0〜v0.1.1の `https://video.local/` 仮想ホスト方式に問題がある可能性が高い。

- `SetVirtualHostNameToFolderMapping` 経由でのHTTP range requestが大容量MP4で正常に機能しない可能性
- moov atom末尾配置（faststart未対応）MP4での仮想ホスト先読み失敗
- Edge自体の file:// 直接アクセスとは読み込みメカニズムが異なる

**v0.1.2 実装方針:**

| 方式 | 説明 | 採用 |
|------|------|------|
| file-uri（主） | player.html を `file://` URI で読み込み、動画も `file://` URI で指定。CORS制約なし、Chromiumのネイティブファイルアクセスを使用 | 主方式 |
| virtual-host（フォールバック） | file-uri方式が失敗した場合に `https://app.local/` + `https://video.local/` 仮想ホストで再試行。player.htmlの再ナビゲーションを伴う | フォールバック |

**方式選択の理由:**

- file-uri方式はChroimiumが大容量ファイルのrange requestをネイティブに処理するため、仮想ホスト方式より安定する可能性が高い
- フォールバック実装により、file-uri方式が予期しない理由で失敗した場合にも旧方式での読み込みを試みる
- 両方式のページコンテキストが CORS 制約により互いに排他的なため、フォールバック時はplayer.htmlを再ナビゲートする

**設計上の制約:**

- 初期版では、プレビューできるMP4を変換対象とする（プレビュー不可の場合はカット位置が指定できない）
- 自動faststart化・プレビュー用一時ファイル生成はv0.2.0以降の検討事項とする
- MP4/AVC(H.264)/AACは引き続き初期版の主対象である

### 4.6 WMPプレビュー（検証）: Windows Media Player COM（v0.4.7）

#### 実装の位置づけ

v0.4.7 で検証目的として `WmpVideoPlayer.cs` を実装した。**標準方式・長期採用方式としては採用しない。**

#### 動作確認済み前提条件

- `wmp.dll`（Windows Media Player COM）は Windows 10/11 に標準インストール済み
- `IWMPPlayer4`（WMPLib GUID: `6BF52A52-394A-11D3-B153-00C04F79FAA6`）で制御

#### 設計上の制約

| 制約 | 内容 |
|------|------|
| 起動時初期化なし | `InitializeAsync` はノーオペレーション。`LoadVideo` 呼び出し時に初めて COM を初期化する |
| ポーリング方式 | COM イベントは購読せず、500ms タイマーで再生位置・状態を取得 |
| 失敗時フォールバック | COM 初期化失敗・読み込みタイムアウト（5秒）時は `VideoError` イベントを発火し、代替プレビューへ切り替え |
| WMP UI 非表示 | `uiMode = "none"` を設定し、WMP の再生コントロール UIを非表示にする |

#### 不採用とした競合技術

| 技術 | 不採用理由 |
|------|-----------|
| LibVLCSharp | DLL 配布の複雑さ（libvlc.dll・libvlccore.dll 等の同梱が必要）・ライセンス管理が課題 |
| WPF MediaElement | `<UseWpf>true</UseWpf>` による起動クラッシュの教訓（v0.3.1 で発生） |
| AxWindowsMediaPlayer（aximp 方式） | SDK スタイルプロジェクトで aximp ラッパーが自動生成されない |

#### AxHost 方式の採用

WinForms の `AxHost` 基底クラスを継承した内部クラス `WmpAxHost` を作成し、`GetOcx()` で `IWMPPlayer4` インターフェースを取得する。`aximp` ツールによる事前ラッパー生成が不要なため、ビルド構成がシンプルになる。

### 4.7 初期版で高度な動画編集機能を対象外とした理由

- 利用者は専門的な動画編集の知識を持たない一般職員である
- 高度な機能（複数区間、音声トラック選択、フレーム単位編集等）は UI の複雑化を招き、操作ミスのリスクを高める
- 初期版では「切り出し」「サイズ縮小」という最低限の需要を確実に満たすことを優先する
- 追加機能は利用状況を踏まえてv0.2.0以降で段階的に検討する

### 4.8 高速カットと圧縮変換の使い分け（v0.1.3改訂）

v0.1.2 までは再エンコード（libx264）のみを提供していたが、長時間動画では処理時間が非常に長くなることが実機確認で判明した。v0.1.3 では「しない（高速カット）」と「圧縮変換」の2種類を提供する。

| 方式 | FFmpegオプション | 速さ | カット精度 | ファイルサイズ |
|------|----------------|------|-----------|---------------|
| 高速カット（初期値） | `-c copy` | 高速（再エンコードなし） | キーフレーム単位（数秒のずれあり） | 元のままの画質・容量 |
| 圧縮変換 | `-c:v libx264 ...` | 低速（全体を再エンコード） | 指定位置に近い精度 | 設定に応じて縮小可能 |

**設計判断:**

- 長時間動画（例: 数時間の会議録画）は再エンコードに非常に長い時間がかかるため、高速カットを初期値とする
- 「開始位置が数秒ずれる」ことより「処理が速い」ことを優先する利用者が多いと判断
- カット精度が必要な場合や圧縮が必要な場合は圧縮変換を選択できる
- 高速カット時は解像度設定は無意味なため UI 上で無効化する（`-c copy` は解像度フィルタを無視する）

## 5. FFmpegコマンド詳細

### 5.1 コマンド構成

**高速カット（しない）:**
```bash
ffmpeg -ss {開始} -to {終了} -i "{入力}" -c copy "{出力}"
```

**速度優先:**
```bash
ffmpeg -ss {開始} -to {終了} -i "{入力}" [-vf {解像度}] -c:v libx264 -crf 28 -preset veryfast -c:a aac -b:a 128k "{出力}"
```

**圧縮変換（画質優先・標準・容量優先）:**
```bash
ffmpeg -ss {開始} -to {終了} -i "{入力}" [-vf {解像度}] -c:v libx264 -crf {品質} -preset medium -c:a aac -b:a {音声ビットレート} "{出力}"
```

### 5.2 品質プリセット

| プリセット | 方式 | CRF | preset | 音声ビットレート | 用途目安 |
|-----------|------|-----|--------|----------------|---------|
| しない（高速カット）（初期値） | `-c copy` | — | — | — | 長時間動画・速さ優先 |
| 速度優先 | 再エンコード | 28 | veryfast | 128k | 再エンコードしたいが処理時間を短くしたい場合 |
| 画質優先 | 再エンコード | 23 | medium | 160k | 高品質な資料動画 |
| 標準 | 再エンコード | 28 | medium | 128k | 庁内共有の一般用途 |
| 容量優先 | 再エンコード | 32 | medium | 96k | メール添付など容量制限がある場合 |

CRF値は数値が小さいほど高品質・大容量。23〜32の範囲は libx264 での実用的な範囲として広く使われており、庁内用途では32でも十分な品質が得られることが多い。

**速度優先と標準の違い:**
- CRF は同じ 28（品質目標は同等）
- preset veryfast はエンコーダが細かい最適化を省くため処理が速い（medium の約 3〜5 倍速）
- veryfast は圧縮効率が medium より低いため、同じ CRF でも出力ファイルがわずかに大きくなる場合がある
- 知覚画質は medium とほぼ同等だが、動きの激しいシーンで差が出る場合がある

### 5.3 解像度プリセット

| プリセット | FFmpegオプション | 備考 |
|-----------|----------------|------|
| 元のまま | （なし） | 入力と同解像度 |
| 720p | `-vf scale=-2:720` | 高さ720px、幅は偶数に自動調整 |
| 480p | `-vf scale=-2:480` | 高さ480px、幅は偶数に自動調整 |

`scale=-2:720` の `-2` は、アスペクト比を保ちつつ幅を偶数に丸める指定。`-1` では奇数幅になり libx264 でエラーになる場合があるため `-2` を採用。

## 6. 出力ファイル命名規則

```
{元のファイル名}_cut_{yyyyMMdd_HHmmss}.mp4
例: meeting_cut_20260523_153000.mp4
```

- タイムスタンプにより同名ファイルとの衝突を防ぐ
- 既存ファイルへの上書きをしない設計（`-y` オプション不使用）
- ファイル名の特殊文字は `_` に置換して安全な文字列にする

## 7. セキュリティ・個人情報

- アプリは個人情報を保存しない
- 外部通信を行わない（WebView2はローカルファイルのみ使用）
- ログはアプリ画面のテキストボックスに表示するのみで、ファイルには保存しない
- 出力ファイルは入力ファイルと同じフォルダに作成するため、利用者がアクセス権を持つ場所のみ出力される

### 4.8 変換時間が長くなる理由と対応方針（v0.1.4整理）

| 方式 | 時間 | 理由 |
|------|------|------|
| 高速カット（-c copy） | 短い（数秒〜数十秒） | 再エンコードなし。ストリームをそのままコピー |
| 圧縮変換（libx264） | 長い | 映像フレームを1枚ずつ再エンコード。30分動画で数分〜十数分 |

**利用者向け推奨:**
- まず高速カットを試す。ファイルサイズが大きすぎる場合や、カット精度が問題な場合のみ圧縮変換を選ぶ
- 動画全体を圧縮変換する場合は、特に時間がかかる（1時間動画で数十分以上）

**将来の検討事項（v0.1.5以降）:**
- `-preset fast` / `-preset veryfast` の採用（処理速度向上、画質はわずかに低下）
- FFmpegの進捗パーセンテージ表示（プログレスバー）
- ハードウェアエンコーダ（`h264_nvenc`、`h264_qsv`）の対応

### 4.9 D&D時のWebView2挙動制御（v0.1.4）

WebView2はChromiumベースのブラウザであり、ファイルをドラッグ＆ドロップするとブラウザ内でその URLへナビゲートしようとする。これにより動画ファイルをアプリにドロップしてもアプリが読み込まず、WebView2内でブラウザ再生されてしまう問題があった。

**対応:** `CoreWebView2.NavigationStarting` イベントで、ナビゲーション先が `.mp4` ファイルの場合に `e.Cancel = true` としてブラウザナビゲーションをキャンセルし、`LoadFile()` で正しくアプリ内に読み込む。

### 4.10 事前変換機能（v0.2.0）

**背景:**

MP4ファイルのメタデータ（moov atom）が末尾に配置されている場合、WebView2によるプレビューが失敗することがある。これはエンコード後に `+faststart` フラグを付けていないMP4に多く見られる。プレビューできないと開始・終了位置を指定できないため、変換実行ができない。

**UX方針:**

利用者には内部処理詳細（faststart・moov atom等の技術用語）を見せない。プレビュー失敗時に「事前変換」として案内し、同意を得てから実行する。

**設計方針:**

| 項目 | 決定内容 |
|------|---------|
| 実行タイミング | 両読み込み方式（file-uri・virtual-host）が失敗した場合、同意ダイアログを表示し、利用者が「実行する」を選んだときのみ実行 |
| 入力ファイル | 元ファイルをそのまま保持（変更・上書きしない） |
| 出力ファイル | `{元ファイル名}_preconverted_{yyyyMMdd_HHmmss}.mp4`（別ファイル作成） |
| 再エンコード | なし（`-c copy` でストリームをそのままコピー） |
| 画質劣化 | なし |
| 事前変換後の動作 | 出力ファイルをプレビューに自動読み込み（`allowPreconvertDialog: false` でダイアログ再案内を抑制） |
| ダイアログ再表示 | ユーザーが手動でファイルを選択し直した場合のみ再度案内する |

**ffmpegコマンド（内部）:**

```bash
ffmpeg -i "{入力}" -c copy -movflags +faststart "{出力}"
```

**利用者向けログ表現:**

- 処理中: 「事前変換中です。大きい動画では時間がかかる場合があります。」
- 完了: 「事前変換が完了しました。変換後の動画を読み込みます。」
- 失敗ダイアログ: 「事前変換に失敗しました。動画ファイルまたは保存先を確認してください。」
- キャンセル: 「事前変換をキャンセルしました。」

**実装上の判断:**

- `FfmpegRunner.RunFaststartAsync` を事前変換処理でも再利用（処理内容は同一）
- プロセス実行コアは `RunProcessAsync` として共通化
- プログレスバー・経過時間タイマー・キャンセル機能は変換処理と同じインフラを再利用
- `_activeOperationLabel` フィールドでタイマーのラベルを動的切り替え（「事前変換中」）
- `_showPreconvertDialog` フラグで事前変換後のファイルに対するダイアログ再表示を防止
- `LoadFile(path, allowPreconvertDialog: false)` により事前変換後の自動読み込みを区別

### 4.11 速度優先プリセット（v0.3.0）

**背景:**

圧縮変換（libx264）は動画全体を再エンコードするため、長時間動画では処理時間が長くなる。v0.2.2 で `SpeedPreset` enum を先行追加していたが、UI への反映は v0.3.0 で実施した。

**設計方針:**

高速カットと速度優先は目的が異なる。利用者が両者を混同しないよう、特徴の違いを UI・マニュアルで明示する。

| 比較項目 | しない（高速カット） | 速度優先 |
|----------|-------------------|---------|
| 再エンコード | なし | あり |
| カット精度 | キーフレーム単位（数秒ずれる場合あり） | 指定位置に近い精度 |
| 解像度変更 | 不可 | 可（720p・480p） |
| 処理速度 | 最速 | 標準より速い |
| 画質 | 劣化なし | 標準と同等〜わずかに低下 |

**FFmpegオプションの選定:**

CRF 28（標準と同じ品質目標）+ preset veryfast を採用した。

- CRF 26 + veryfast も検討したが、「速度優先」は速度のトレードオフであり品質を上げる意図ではないため不採用
- CRF 28 + veryfast は「標準の品質目標を維持しつつ、preset を veryfast にして処理速度を上げる」という明確なセマンティクスを持つ
- 出力ファイルサイズは標準（medium）よりわずかに大きくなる場合があるが、同じ CRF のため知覚画質は概ね同等

**UI 方針:**

- ComboBox の表示順: しない（高速カット）→ 速度優先 → 画質優先 → 標準 → 容量優先
- 速度優先選択時のヒント: 「速度優先は、変換時間を短くするための設定です。画質は標準より少し低下する場合があります。」
- 解像度コンボは速度優先時も有効（他の圧縮変換と同様）
- 変換ログ: 「出力方式: 圧縮変換（速度優先）」

### 4.12 代替プレビュー方式: WPF MediaElement（v0.3.1）

**背景:**

標準プレビュー（WebView2）は Windows 10 20H2 以降・Windows 11 で動作するが、一部の端末環境・一部の MP4 ファイルでプレビューが失敗する場合がある。v0.3.1 では WPF MediaElement を代替プレビューとして追加し、標準プレビューが失敗した場合のフォールバック手段を提供する。

**実装方式:**

| 項目 | 内容 |
|------|------|
| コントロール | `System.Windows.Controls.MediaElement`（WPF） |
| WinForms 埋め込み | `System.Windows.Forms.Integration.ElementHost` で WinForms Panel に埋め込む |
| プロジェクト設定 | `<UseWpf>true</UseWpf>` を csproj に追加（`WindowsFormsIntegration` が自動参照される） |
| 位置検出 | WPF MediaElement に PositionChanged イベントなし。WinForms Timer（250ms）でポーリング |
| LoadedBehavior | `Manual`（Play() 呼び出しで読み込み開始） |

**読み込みトリガー方式（`_loadingOnly` パターン）:**

WPF MediaElement の `LoadedBehavior = Manual` では、`Source` を設定しただけでは `MediaOpened` が発火しない。`Play()` を呼ぶことで読み込みが開始され `MediaOpened` が発火する。

- `LoadVideo()` 内: `Source` 設定 → `Play()` で読み込みトリガー
- `OnMediaOpened()` 内: 即座に `Pause()` → 動画を停止してからユーザーに委ねる
- この `Pause()` 呼び出し中は `_loadingOnly = true` のため `PlaybackPaused` イベントが UI に伝播しない

**フォールバックフロー:**

```
標準プレビュー（WebView2）読み込み失敗
  ↓
「標準のプレビューで再生できませんでした。代替プレビューで開き直しますか？」
  ↓ YesNo ダイアログ（自動切り替えなし、利用者の同意を取得）
  ↓ Yes
代替プレビュー（WPF MediaElement）で同一ファイルを読み込み
  ↓ さらに失敗した場合
事前変換ダイアログ（既存のフォールバック）
```

**UI: プレビュー方式コンボボックス:**

画面右上の「プレビュー方式:」コンボボックスで手動切り替えも可能。
- 「標準」: WebView2（file-uri + virtual-host フォールバック）
- 「代替」: WPF MediaElement（ElementHost）

**環境依存の注意:**

WPF MediaElement は端末にインストールされている .NET WPF ランタイム・コーデックに依存する。再生できるファイルは端末環境によって異なる場合がある。

**LibVLCSharp を採用しなかった理由:**

| 比較項目 | WPF MediaElement | LibVLCSharp |
|---------|-----------------|-------------|
| 外部DLL | 不要（.NET 標準） | libvlc*.dll の配置が必要 |
| 配布容易性 | ○ | △（DLL一式の同梱が必要） |
| 閉域環境での配布 | 問題なし | DLL管理・ライセンス確認が必要 |
| 初期版での採用 | 採用 | 今回は採用しない |

LibVLCSharp は再生対応形式が広く安定しているが、libvlc DLL の配置・ライセンス確認が必要となる。閉域環境への配布容易性を優先し、v0.3.1 では WPF MediaElement を採用した。

### 4.13 代替プレビュー方式: ffmpeg サムネイル生成（v0.4.3）

**背景:**

v0.3.1 で WPF MediaElement を代替プレビューとして実装したが、`<UseWpf>true</UseWpf>` による `DllNotFoundException` が起動時クラッシュを引き起こしたため v0.3.2 で無効化した。v0.4.3 では WPF を使わず、ffmpeg でサムネイル静止画を生成する方式で代替プレビューを再実装した。

**選定理由:**

| 比較項目 | WPF MediaElement（v0.3.1） | ffmpeg サムネイル（v0.4.3） |
|---------|--------------------------|--------------------------|
| 起動安定性 | △（UseWpf=true でクラッシュ発生） | ◎（WinForms のみ） |
| 動画再生 | ○（リアルタイム再生可能） | ✗（静止画のみ） |
| カット位置確認 | ◎（再生しながら確認） | ○（コマ送りで確認） |
| 依存性 | .NET WPF ランタイム + コーデック | ffmpeg.exe のみ |
| 実装複雑度 | 高（ElementHost・WPF 相互運用） | 低（Process + PictureBox） |

**実装方式:**

`FallbackPreviewPanel` クラス（`Panel` サブクラス）として実装。`MainForm` の `_previewSlot` コンテナに WebView2 プレビューと並べて配置し、Visible の切り替えで表示を制御する。

**ffmpeg コマンド:**

```bash
ffmpeg -ss HH:MM:SS.mmm -i "{入力}" -frames:v 1 -q:v 2 -y "{TEMP}\MovieConverter_preview\thumb.jpg"
```

`-ss` を `-i` の前に指定することで高速なキーフレームシーク（デコードコスト小）を実現する。`-q:v 2` は JPEG 品質（最高品質に近い値）。

**キャンセル方式:**

ナビゲーション操作が連続した場合、前のサムネイル生成をキャンセルして新しいリクエストを優先する。`CancellationTokenSource` の置き換えパターンで実装。

**フォールバックフロー（v0.4.3）:**

```
WebView2 InitFailed = true（未インストール）
  ↓
FallbackPreviewPanel を _previewSlot に追加し WebView2 を非表示に
  ↓
ffmpeg でサムネイル生成 → PictureBox に表示
  ↓ ナビゲーションボタン / 時刻テキストボックスで位置移動
  ↓ TimeChanged イベント → MainForm で _currentTime 更新
  ↓ 「現在位置を開始に設定」「現在位置を終了に設定」でカット範囲確定
  ↓
変換実行（既存の変換フローをそのまま使用）
```

**設計上の注意:**

- WPF MediaElement は採用しない（起動クラッシュの教訓）
- LibVLCSharp は採用しない（DLL 配布の複雑さ）
- ffmpeg.exe が未配置の場合はサムネイルを生成しないが、テキスト入力による位置指定と変換は可能
